❤️ the web for what it is

Hello, beautiful people! How are we doing so far? Good? Firstly, I want to say thank you to our awesome speakers for blowing our minds with their wonderful talks. I don't know about you, but I sure learned a tonne from them.

And also, to each and every one of you, you have no idea how much we appreciate you for choosing to spend your Wednesday afternoon with us. 3pm probably wasn't our best decision, but hey, you're all here now. So I'm going to be wrapping up the evening with a talk called “Love the web for what it is.”

Can most of you remember your first encounter with the web? I can. We used a dial-up modem, you know, the kind where if you connected to the internet, nobody could call the house line?

Back then, butlers were in style. Online chat programs had meaningful acronyms, and people made websites about things they liked. Google didn't exist and I still have the same email account I got when I was 11.

There was something called the Browser Wars, when browsers competed by adding non-standard features and extensions to the fledgeling HTML, CSS and Javascript syntax. I did not fight in said war, I was merely part of what was being fought over.

In fact, when I got my first “real” job as a web developer, fixed layouts were already falling out of favour, and responsive design was the norm. That was great, because when it came to web design, I developed a mental model of the web that was dynamic from the get-go. I didn't really have to overhaul my way of thinking when designing for the web.

Now, according to my friends who do not build websites for a living, this behaviour is not normal. And I'll just look at them like, what do you mean you don't resize your browser 1000 times a day?

But jokes aside, in my mind, the web is a dynamic and interactive medium. And it was upon learning more about browsers’ layout engines, how rendering works and their relationship with HTML and CSS that has shaped my current opinion of web design.

“If you don’t know where you’ve come from, how will you appreciate where you’re going?”
–Shashi Tharoor

Perhaps you may not agree with me, but I do believe that an understanding of the history of the web, amongst many other things, is essential to becoming a better web designer. Let’s be honest, the digital age we live in now moves at a break-neck speed. The first electronic computers came about in the 1940s, that’s less than a century ago.

Which is why I think it is important for us to understand how the web came to be in its current state today. To give us a greater sense of appreciation of what we have right now, and inform a direction of where we ought to be going, moving forward.

Basic building blocks of the web

We often talk about this holy trinity of HTML, CSS and Javscript when it comes to the web. HTML for structure, CSS for style and Javascript for interactivity.

All 3 are key aspects of the web, and all 3 have evolved greatly since their inception.

HTML (Hypertext Markup Language)

Information Management: A Proposal by Sir Tim Berners-Lee

In 1989, Sir Tim Berners-Lee proposed a solution to the problem that CERN had, with regards to information access, and his paper introduced the idea of linked information.

This concept of cross-referencing information within documents could allow for a “web” of information to be held electronically on machines across the world, via some form of hypertext.

Sir Tim Berners-Lee is credited with inventing the World Wide Web because he developed the language for encoding documents (HTML), the protocol for linking them (HTTP) and the method of addressing them (URL). The web was built for the publication and retrieval of information.

CSS (Cascading Style Sheets)ss

Cascading HTML style sheets -- a proposal by Håkon Wium Lie

The story of CSS started in 1994. You see, the web started to become more popular as a publication medium, but there was no good way to style documents at the time. Håkon Wium Lie saw a need to have a stylesheet language for the web, and presented his proposal for CSS at the 2nd International World Wide Web conference in Chicago.

Stylesheets in browsers were not a new idea, as separation of document structure from presentation had been a goal of HTML since the beginning. Sir Tim Berners-Lee wrote his NeXT browser/editor to support use of a simple style sheet to determine document styles.

But he decided to let browsers decide for themselves how to best display pages to their users. A very laissez-faire approach. So users of the web did what users do best. Complain. Largely about how they couldn't control how their pages looked.

Proposals are destined to remain mere academic endeavours, if they are not implemented into real-world applications.

The first commercial browser to support CSS was Microsoft's Internet Explorer 3, followed by Netscape Navigator 4.0, and then Opera 3.5.

Javascript

The Origin of Javascript with Brendan Eich on the Javascript Jabber podcast

Javascript's origin story started in 1995, when Netscape wanted a way to make the web experience more dynamic. They hired Brendan Eich as a contractor to implement Scheme, which was a Lisp dialect, in the browser.

In his words, the idea was to make something that Web designers, i.e. people who may or may not have much programming training, could use to add a little bit of animation or a little bit of smarts to their Web forms and their Web pages.

So in 1996, JavaScript was implemented in Netscape first, then Internet Explorer. Fun fact, the IE version was actually called JScript, because trademarks.

When Sir Tim Berners-Lee created the web, he imagined the web as an open platform that would allow everyone, everywhere to share information, access opportunities and collaborate across geographic and cultural boundaries. That's why the web was released to the public domain back in 1993.

Mozilla brings Microsoft, Google, the W3C, Samsung together to create cross-browser documentation on MDN

So the Browser Wars were a dark time for the web, but in its aftermath, something good emerged: web standards. Which is an effort to keep the web open, interoperable and accessible.

We have come a long way in the relatively short span of two decades. From browsers trying to one-up each other with non-standard implementations, to working together to create technical documentation for web developers, built on an open-source web development documentation platform based on wiki technology.

Blue Beanie Day

November 30

“Web standards play an important role when it comes to web accessibility. It is important that websites can be displayed with any browser.”

Some of you might be thinking, but browsers are still broken, right? I'm going to make the assumption that a number of people in this room have had the experience of building a comp in one browser that works great, only to find that it's rendering weird in another one (one that you obviously did not test at all during the build process).

What we need to understand is that browsers are software, software that is built by human engineers, working for different companies. Each of these companies have their own internal dynamics and decision-making matrices that outsiders like us are simply not privy to. So no, features will not be universally supported all at the same time.

But because of web standards, they will get there.

The Web Standards Project was formed in 1998, essentially to tell browser vendors to quit it and stop messing up the web. And they were successful. Browser makers are no longer the problem.
“The problem lies with designers and developers chained to the browser–quirk–oriented markup of the 1990s—often because they don’t realise it is possible to support current standards while accommodating old browsers.”
—WaSP (Web Standards Project)
The onus now falls onto us, designers and developers of the web, to make sure that the sites we build today will work for all, today and tomorrow. That means keeping browser-specific markup out of the code we write, and building with web standards in mind.

Most of the information on the web is consumed via an electronic screen. Electronic screens were introduced as an output device in the 1950s and screen technology has improved rapidly over these past few decades.

But electronic screens are a proxy medium, in that whatever we see displayed on a screen comes from data encoded as electronic signals. The heart of electronic displays is light. It's not tangible. We can't touch light.

Because the web is a completely new medium, it's only natural that people try to reconcile it with something familiar, like print. Even the terminology was borrowed from print, we use terms like web pages.

However, there is an additional degree of separation between the creator and the final output. As such, we cannot have the same set of expectations when it comes to handling the web as a medium.

I think this emoji explanation sums it up reasonably well.

Designing for the web requires an intimate knowledge of the browser that will be rendering the final output. You can’t stick your hands in there and change things directly. You have to change the instructions given to the browser, explaining how you would like things to be rendered instead.

When browsers first started out, they couldn’t do much except display text and maybe images if you were lucky. Like a baby, that hasn’t developed the motor skills to even sit up yet.

But over the years, browsers have become more mature and vastly more capable. And when it comes to layout on the web, I feel that the release of CSS grid last year is like the web finally getting it’s driver’s license. It's like the web has come of age, you know?

There are still a lot of legacy expectations brought over from print that cause designers a lot of frustration when designing for the web. I know, because I’d given my fair share of grief to them when I said, no, we cannot build that because it requires me resorting to measures of extreme-hackishness. So hell, no!

A new normal for the web

Which is why I want to establish a new normal for the web. A normal where people don't expect websites to look the same on different browsers and devices. In this new normal, we embrace the fluidity of content and work with it instead of against it. This is a normal where we cede control of our designs to the browsers that render them.

New layout techniques like Flexbox and CSS grid give us a whole suite of commands to tell the browser how we’d like things to look depending on the context of the screen being used. And it’s much easier to trust the browser if we understand how it works.

and many more...

I know there are a boat-load of browsers using different layout engines and hence each have their own quirks, or release features at different times, and you know what? That’s perfectly fine. Your parents didn’t expect you to behave exactly like your siblings, did they? If they did, I'm sorry, no judgement.

Embrace feature queries

.selector {
  /* Styles that are supported in old browsers */
}

@supports (property:value) {
  .selector {
    /* Styles for browsers that support the specified property */
  }
}

As builders for the web, we need to adapt to the evergreen browser model where features and bug fixes get shipped rapidly, so we layer on different styles and looks that are not browser-specific, but rather, feature-specific.

Feature queries are a fantastic addition to our toolbox, because they allow us to perform this layering natively, without resorting to extraneous code libraries. If you had to write extra code to make the website look exactly the same on every browser, you could write less code and have designs that work well on browsers with different levels of feature support.

I've come to the realisation that designing layouts on the web requires interpolative thinking, on multiple levels. We don't just think in one fixed dimension, we get to think about how our design will morph on a narrow screen, or on an older browser, in addition to how it will appear on a browser with the latest features.

There is no other medium that let's us do this. How great is that? To me, our medium is really something special.

I reuse this example all the time, so maybe you've already seen this before, but the point I'm trying to make here is that, we need to move away from the idea that a site or page needs to look the same on every browser.

CSS shapes is one of my favourite properties. You know I'm a CSS nerd because I actually can rank CSS properties based on personal preference. But anyway, CSS shapes is one of those properties with rather uneven support. And hence a great candidate for feature queries.

IE11, and any older browser that does not support feature queries, will still get styled, so the content can be comfortably consumed. A newer browser like Firefox, which does not support CSS shapes yet, gets the visual treatment in the middle, but once CSS shapes ships, it will automatically look like the design on the right. Current Chrome users see that now, because CSS shapes has been in Chrome for a while already.

We do not have to go back and change our code every time the browser updates itself. In a way, it is now much safer than before to use cutting edge CSS in your projects, without having to worry that things will be broken on older browsers.

“CSS isn't a programming language. It's a stylesheet language. We shouldn't expect it to behave like a programming language. It has its own unique landscape and structures, ones that people with programming language mental maps might not expect.”
—Danielle Huntrods

Danielle Huntrods from Clearleft said it best. CSS has its own unique landscape and structures, and our mental maps around it need to reflect that.

This is anecdotal, but my observation, based on a very limited sample size, is that it is often those who write the best back-end code, whose mental models around traditional programming paradigms are the strongest, who find CSS the most uncomfortable.

When you think about CSS, think about water, and how it flows. We can learn to channel water to our benefit, or fight it and drown. Okay, maybe that was a little too dramatic, but you get my point, right?

CSS is ultimately a holistic technology, in that, even though you can use properties in isolation, the full power of CSS shines through when used in combination. CSS is a team sport.

For example, to do layout, we usually start off with using the display property. But eventually, most of us end up using a whole suite of properties that number more than players on a basketball team. These properties work together to make magic on the web.

“The hardest part is changing our thinking, not our CSS.”
—Jen Simmons

We can always recreate the same old boring layouts that we've always built with these new powerful tools at our disposal. Or we can draw inspiration from graphic design and print magazines, from film, even architecture. These sources that inspire designs which, only a year before, I was saying could not be built. They couldn’t be built before, but they can be now.

Some of you be thinking, why am I asking you to reinvent the wheel? Aren't we supposed to be building things faster, with less effort, ship ship ship! I don’t support that view. Here's my counter-question, is the wheel still relevant when you’re no longer on land? The web is a unique medium altogether. Maybe we ought to be thinking about wings instead.

It’s not that I want things to take forever to build, but I still maintain a moderately romantic view of craftsmanship. Like the potter who works and shapes his clay by hand, like the sculptor who chisels his marble with the precision of a surgeon, I would like to direct my browser like a conductor does his orchestra.

Shape the web we use and build for

  • CSS specifications are all on GitHub
  • Raising browser bugs can and does make a difference
  • Write and talk about CSS features
  • Browser developers are people too, provide feedback and be nice about it.

To me, the magic of the web is the process by which the web evolves. The web was meant to be an open platform. One where new features can be proposed and discussed by the community before being implemented into browsers.

And so, I want to encourage all of you to participate in shaping the web platform. You may think your voice is just a drop in the ocean, but it is not, it is significant. All the CSS working drafts are hosted on GitHub and we are free to raise issues for clarification or even submit pull requests if we spot any errors.

Raising browser bugs sends a signal to browser vendors that certain features are in use, prompting them to fix those bugs sooner rather than later. Conversely, if we don't use a feature simply because it is buggy, browser vendors would think that feature is not in demand and prioritise something else instead.

Writing and talking about certain CSS features provides visibility and feedback to browser vendors and specification writers as well. But regardless of platform, remember that browser engineers and specification writers are human too, and we all have feelings. So be kind, the world needs more kindness. And it is only through our collective effort, that we can make the web a better place for everyone.
